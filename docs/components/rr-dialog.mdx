## Overview

Use `rr-dialog` for a blocking decision or focused task that must interrupt the page — confirmations, destructive-action checks, small focused forms. It's built on the native `<dialog>` element via `showModal()`, so focus trap, Escape-to-close, and `::backdrop` come from the platform, not reimplementation. Open it with `show()` (or the `open` attribute); every close path — Escape, the close button, backdrop click (`close-on-backdrop`), programmatic `close()` — funnels through one code path and fires a single `rr-dialog-close`. If the user doesn't need to be interrupted, don't use a dialog: inline `rr-alert` or a dedicated page respects flow better.

## Examples

### Confirmation dialog with footer actions

```html
<rr-dialog heading="Confirm action" open>
  Are you sure you want to delete this item?
  <rr-button slot="footer" variant="danger">Delete</rr-button>
  <rr-button slot="footer" variant="secondary">Cancel</rr-button>
</rr-dialog>
```

### Body-only dialog

```html
<rr-dialog heading="Details">
  <p>Dialog body content goes here.</p>
</rr-dialog>
```

### Non-dismissible backdrop

```html
<rr-dialog heading="Processing" close-on-backdrop="false" open>
  Please wait while we finish.
</rr-dialog>
```

## Accessibility

- **WAI-ARIA pattern:** [www.w3.org/WAI/ARIA/apg/patterns/dialog-modal/](https://www.w3.org/WAI/ARIA/apg/patterns/dialog-modal/)
- **Focusable:** yes
- **Keyboard:** Escape closes the dialog; Tab and Shift+Tab cycle focus within the dialog (native modal focus trap via showModal()); Enter or Space activates the close button when focused
- **WCAG:** 1.4.3 Contrast (Minimum), 2.1.1 Keyboard, 2.1.2 No Keyboard Trap, 2.4.3 Focus Order, 4.1.2 Name, Role, Value

**Rules**

- Provide a heading so the dialog has an accessible name
- The close button always carries aria-label="Close dialog"

## Usage

- **Do** funnel every dismissal through the component (`close()`, Escape, backdrop) and listen for the single `rr-dialog-close` event to clean up state.
- **Do** keep the footer to one primary action plus a ghost/secondary cancel — the one-primary rule applies inside dialogs too.
- **Don't** disable `close-on-backdrop` and remove the close button together unless the flow truly cannot be abandoned (destructive confirmations may; forms should not).
- **Don't** nest dialogs; if a dialog needs a second dialog, the task has outgrown a modal.
