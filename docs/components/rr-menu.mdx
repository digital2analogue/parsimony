## Overview

`rr-menu` is the dropdown action menu — a trigger control that opens a floating list of commands (WAI-ARIA menu-button pattern). Slot any control into `trigger` and `rr-menu-item` elements into the body; the component owns everything hand-rolled menus routinely get wrong: trigger ARIA wiring (`aria-haspopup` / live `aria-expanded`), full arrow-key navigation with wrapping and disabled-item skipping, Home/End, Escape-with-focus-return, Tab-closes, and outside-click dismissal.

Reach for it when the user picks an **action** (edit, duplicate, delete). If they're picking a **value** that fills a form field, that's a select/listbox (`rr-select`) — a menu that mutates state on arrow-through is the classic misuse. If the action needs confirmation or input, the menu item should open an `rr-dialog` rather than grow controls of its own.

The popup positions itself below the trigger with plain CSS — no portal — aligned to the start or end edge via `placement`. That keeps the component dependency-free, with two contracts: don't trap it inside an `overflow: hidden` ancestor, and know that the popup never auto-flips — near the viewport bottom, opening upward is the consumer's call (there is no top placement yet; scroll the trigger into view or keep menus out of the last screen rows).

## Accessibility

- **WAI-ARIA pattern:** [www.w3.org/WAI/ARIA/apg/patterns/menu-button/](https://www.w3.org/WAI/ARIA/apg/patterns/menu-button/)
- **Focusable:** no
- **Keyboard:** Enter, Space, or click on the trigger toggles the menu; ArrowDown on the trigger opens and focuses the first item; ArrowUp opens focusing the last; ArrowDown/ArrowUp move through enabled items and wrap; Home/End jump to the ends; Enter or Space activates the focused item; Escape closes and returns focus to the trigger; Tab closes and moves on
- **WCAG:** 1.4.3 Contrast (Minimum), 2.1.1 Keyboard, 2.4.3 Focus Order, 4.1.2 Name, Role, Value

**Rules**

- The slotted trigger receives aria-haspopup=menu and live aria-expanded automatically
- Disabled items are skipped by keyboard navigation and are inert to clicks
- Focus returns to the trigger on Escape and on item selection

## Usage

**Do** always set `label` — it's the accessible name of the popup ("Row actions", "Ruleset options"). A menu without a label is anonymous to a screen reader.

**Do** listen on `rr-menu-select` and switch on `detail.value` — one handler per menu, not one per item.

**Do** give icon-only triggers their own accessible name — `label` names the popup, not the trigger. A "⋯" ghost button needs an `aria-label` of its own (e.g. `aria-label="Row actions"` on the slotted rr-button).

**Do** put the menu's containing element above clipping contexts. The popup is CSS-positioned, so an `overflow: hidden` or `overflow: auto` ancestor will crop it — the table-cell overflow menu is exactly where this bites.

**Don't** use a menu for value selection (choosing an operator, a status, a data element) — that's `rr-select`'s job. Menus fire actions; selects fill fields.

**Don't** exceed roughly seven items. Past that, group with a second menu, promote frequent actions to visible buttons, or rethink the surface.

**Don't** put more than one `danger` item in a menu, and keep it last — destructive actions live at the bottom, isolated from the muscle-memory path. (There is no separator element yet; when one lands it should draw with `--color-border-muted`, the separator-within-a-surface role.)
