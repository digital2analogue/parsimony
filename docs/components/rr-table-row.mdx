## Overview

`rr-table-row` is a row inside `rr-table` — like `rr-tab` inside `rr-tab-list`, it exists only in that context. The host carries `role="row"` and lays out as a CSS table row; its visual states (zebra fill, hover tint, selected tint + rail) are painted by the parent table's stylesheet so every table in the system shares one recipe.

The row owns exactly one piece of state: `selected`. Set it from your selection model and the row exposes `aria-selected="true"` and receives the strongest visual treatment — the quiet green tint plus the solid 3px action rail. Everything else (hover, zebra) is the table's business.

## Usage

**Do** drive `selected` from a single selection model in your app — the row reflects state, it doesn't manage it.

**Do** keep every row's cell count equal to the header's — CSS table layout will happily render a ragged grid, and it reads as broken.

**Don't** attach click handlers to rows in a table that isn't marked `interactive` — users get no hover affordance that the row is clickable.

**Don't** try to style row states from outside — hover, zebra, and selected recipes live in `rr-table` so tables stay uniform. If a state is missing, add it to the system, not to one table.
