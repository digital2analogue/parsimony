## Overview

`rr-table` is the structural data table ‚Äî the visual language for rows-and-columns data across the system: muted all-caps column headers, quiet `border.muted` row separators, flat corners (`radius.none`), no card chrome. It composes from `rr-table-row` and `rr-table-cell`, and cells take arbitrary content ‚Äî badges, menus, buttons ‚Äî which is what real tables hold.

A note on the architecture: raw `<tr>`/`<td>` can't be slotted into a custom element (the HTML parser drops table tags outside a `<table>` context), so the family uses CSS table display with explicit ARIA table roles ‚Äî the host is `role="table"`, rows are `role="row"`, cells `role="cell"`/`role="columnheader"`. The semantics live on the hosts, so the accessibility tree reads correctly through the shadow boundary.

Row states are owned here, not by rows: `zebra` stripes even rows with the elevated surface, `interactive` enables the green hover tint (accent green signals interactivity ‚Äî a table that doesn't respond to clicks doesn't get the tint), and a row's `selected` attribute paints the tint plus the solid action rail. Rows emit no events of their own ‚Äî on an `interactive` table, attach your own click handler and ignore clicks that originate inside interactive cell content (`event.target.closest('rr-menu, rr-button, a')`). Neither the table nor its cells clip overflow, so CSS-positioned popups like `rr-menu` render fine inside cells (mind any scrolling wrapper *you* add around the table). Reach for `rr-table` for read-and-scan data; if you need editing, sorting, or drag-reorder, build that behavior on top ‚Äî the table stays presentational.

## Accessibility

- **WAI-ARIA pattern:** [www.w3.org/WAI/ARIA/apg/patterns/table/](https://www.w3.org/WAI/ARIA/apg/patterns/table/)
- **Focusable:** no
- **Keyboard:** Interactive content inside cells (menus, buttons, links) keeps its own keyboard behavior; the table adds none
- **WCAG:** 1.3.1 Info and Relationships, 1.4.3 Contrast (Minimum), 4.1.2 Name, Role, Value

**Rules**

- The host carries role=table with aria-label from the label prop; shadow rowgroups carry role=rowgroup
- Because layout is CSS tables (not <table>), the explicit ARIA roles on host/rows/cells are the semantics ‚Äî do not remove them
- Selected rows expose aria-selected=true

## Usage

**Do** always set `label` ‚Äî it's the table's accessible name ("Decision rules", "Applications"). An unlabeled table is a mystery grid to a screen reader.

**Do** use `density="compact"` for data-dense operational surfaces and the default for editorial contexts ‚Äî pick per surface, not per table.

**Do** mark amount/count/ID columns `numeric` on both the header and body cells so the right-alignment lines up.

**Don't** set `interactive` on a table whose rows don't respond to clicks ‚Äî the hover tint is a promise of interactivity, and a broken promise is worse than no affordance.

**Don't** combine `zebra` and `interactive` casually: zebra's elevated fill and the hover tint are both quiet, and stacked they muddy each other. Dense reference tables get zebra; clickable row lists get interactive.

**Don't** reach for this component for layout ‚Äî it's for data. Page structure belongs to CSS grid/flex.

**Don't** leave a control column's header empty ‚Äî a column of overflow menus still gets a real header label ("Actions").

A note on `aria-selected`: strictly, ARIA reserves it for rows inside `grid`/`treegrid`. This family keeps `role="table"` (no grid keyboard-navigation contract is implemented) and accepts the pragmatic attribute ‚Äî screen readers that ignore it lose nothing, since selection is also visible in the accessibility tree via the row's state. If a true grid interaction model lands later, the role switches with it.
