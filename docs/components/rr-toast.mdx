## Overview

`rr-toast` is the transient half of the feedback family: a floating notification that announces the outcome of an action, lingers briefly, and leaves. Where `rr-alert` is an in-flow tinted banner that belongs to the page (it stays until the condition it describes is resolved), the toast is a neutral elevated surface — `background.alt` + `shadow.dialog` — that overlays the page and expresses its variant through a 3px accent rail and heading color only, so a stack of mixed-variant toasts still reads as one family.

Reach for a toast when the user just did something and needs confirmation or a recovery path (saved, deleted + undo, upload failed). Reach for `rr-alert` when the message describes a persistent state of the page itself (a form with errors, a degraded service notice). If the message requires a decision before the user can continue, neither is right — use `rr-dialog`.

The component owns the surface, the status semantics (`role="status"`, escalating to `role="alert"` for danger), the optional auto-dismiss timer (paused while hovered or focused, per WCAG 2.2.1), and the dismiss affordance. Screen readers announce the toast's rendered text — heading plus message — when it appears, politely by default and assertively for danger. Placement is deliberately the consumer's job: render toasts inside your own `position: fixed` region (typically a bottom corner, inset by `--spacing-element`) and stack them with `--spacing-tight`. The region itself needs no ARIA role — each toast is its own live region. To close a toast from code (e.g. after its Undo ran), call `dismiss()`; the reason reported is `"programmatic"`.

## Examples

### Success

```html
<rr-toast variant="success" heading="Saved">Your changes are live.</rr-toast>
```

### Danger, persistent

```html
<rr-toast variant="danger" heading="Upload failed">The file could not be processed.</rr-toast>
```

### With undo action

```html
<rr-toast heading="Rule deleted">
  "High-value approvals" was removed.
  <rr-button slot="action" variant="ghost" size="sm">Undo</rr-button>
</rr-toast>
```

## Properties

| Prop | Type | Default | Description |
|------|------|---------|-------------|
| `variant` | `string` | `neutral` | Status variant: neutral (default), success, warning, danger, info. Expressed via the accent rail and heading color. |
| `heading` | `string` | `` | Optional bold heading rendered above the message, in the variant accent color. |
| `duration` | `number` | `0` | Auto-dismiss after this many milliseconds. 0 (default) means persistent until dismissed. Timer pauses while hovered or focused. |
| `dismissible` | `boolean` | `true` | Shows the dismiss button. On by default — a toast the user can't clear is a trap. |

## Accessibility

- **WAI-ARIA pattern:** [www.w3.org/WAI/ARIA/apg/patterns/alert/](https://www.w3.org/WAI/ARIA/apg/patterns/alert/)
- **Focusable:** no
- **Keyboard:** Tab moves focus to the action and dismiss button; Enter or Space activates the focused control; The auto-dismiss timer pauses while the toast is hovered or holds focus
- **WCAG:** 1.4.3 Contrast (Minimum), 2.2.1 Timing Adjustable, 4.1.3 Status Messages

**Rules**

- role=status with aria-live=polite by default; the danger variant escalates to role=alert / assertive
- The dismiss button carries an aria-label of "Dismiss"
- Never put the only path to an action inside an auto-dismissing toast — duration must be 0 when an action slot is used, or the action must exist elsewhere

## Usage

**Do** pair every destructive action with a toast carrying an Undo action in the `action` slot — capture the removed item and its position, restore it in the Undo button's click handler, then call the toast's `dismiss()` method (which fires `rr-toast-close` with `detail.reason: "programmatic"`). Never restore from inside the `rr-toast-close` handler — the event fires for every close, including plain dismissals.

**Do** keep toasts persistent (`duration="0"`, the default) when they carry an action. An auto-dismissing toast whose Undo is the only recovery path is a timing trap — if you must auto-dismiss, the action needs to exist somewhere else too.

**Do** give variant toasts a heading. The accent rail alone is a color-only signal; the heading carries the meaning for everyone else.

**Don't** use a toast for validation errors on a form the user is still editing — that's a persistent page state, which is `rr-alert`'s job (or inline field errors).

**Don't** stack more than three toasts; collapse older ones or summarize. A wall of toasts is a log, not feedback.

**Don't** position the toast from inside the component (no `position: fixed` overrides on the host). The consumer's toast region owns placement, stacking order, and enter/exit motion.
