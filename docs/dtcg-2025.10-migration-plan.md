# DTCG 2025.10 Migration Plan — Phase 1 (value-format compliance)

**Status:** proposed — awaiting review. No token files changed yet.
**Scope:** Phase 1 only. Bring token *value encoding* up to DTCG 2025.10 and the
tooling needed to build it. **Resolver module and `$deprecated` are explicitly
out of scope** (deferred to a later phase).
**Guiding constraint:** the built CSS in `build/css/` must not change. Consumers
(`@digital2analogue2/tokens`, `portfolio-vercel`) are insulated and must stay so.

> This is a migration plan, not a decision entry. The decision itself belongs in
> `docs/decisions.md` once we commit to it (one decision log — see CLAUDE.md).

---

## 1. Why

The DTCG spec reached its first stable version, **2025.10** (2025-10-28), and
split into Format / Color / Resolver modules. Several scalar token values moved
from strings to structured objects. Our source still uses the pre-stable string
forms. Phase 1 closes that gap for everything except theming (resolver).

- Spec: <https://www.designtokens.org/tr/drafts/format/>,
  <https://www.designtokens.org/tr/drafts/color/>
- Announcement: <https://www.w3.org/community/design-tokens/2025/10/28/design-tokens-specification-reaches-first-stable-version/>
- Tooling: Style Dictionary v5 adopted 2025.10 as its base format.

## 2. What changes (value encoding)

| Type | Count | Today | 2025.10 target |
|---|---|---|---|
| color | 62 leaf | `"#ECFDF5"` | `{ "colorSpace": "srgb", "components": [r,g,b], "alpha": a, "hex": "#ECFDF5" }` |
| dimension | 44 (22 px / 8 rem + group nodes) | `"2px"`, `"0.625rem"` | `{ "value": 2, "unit": "px" }`, `{ "value": 0.625, "unit": "rem" }` |
| duration | 5 (all ms) | `"120ms"` | `{ "value": 120, "unit": "ms" }` |
| shadow | 5 | sub-values are strings (`"0"`, `"1px"`, `"rgba(0,0,0,0.08)"`) | offsetX/offsetY/blur/spread → dimension objects; `color` → color object |
| typography | 21 | composite of `{ref}` aliases | **no edit** — inherits via the dimension/weight primitives it references |
| transition | 4 | composite of `{ref}` aliases | **no edit** — inherits via duration/cubicBezier |
| cubicBezier / fontFamily / fontWeight / number | 17 | as-is | **no change** |

### Transformation rules (deterministic — no design intent changes)

- **color:** keep the existing hex as the authoritative input. `colorSpace` is
  `"srgb"` for every current token (the whole palette is sRGB hex — **no gamut
  expansion in Phase 1**). `components` are the sRGB channels normalised to
  0–1, rounded to a fixed precision (proposed: 4 dp). `hex` is retained as the
  6-digit fallback. `alpha` only emitted when not 1.
  - `#ECFDF5` → `{ "colorSpace": "srgb", "components": [0.9255, 0.9922, 0.9608], "hex": "#ECFDF5" }`
- **dimension:** split numeric + unit. Unit is `px` or `rem` (both already in
  use; both allowed by 2025.10). Unitless `0` → `{ "value": 0, "unit": "px" }`.
- **duration:** `"120ms"` → `{ "value": 120, "unit": "ms" }`.
- **shadow sub-values:** each offset/blur/spread becomes a dimension object;
  the `rgba(...)` color becomes a color object with `alpha` carried across
  (`rgba(0,0,0,0.08)` → `{ "colorSpace": "srgb", "components": [0,0,0], "alpha": 0.08, "hex": "#000000" }`).

A throwaway codemod (`scripts/_migrate-dtcg-2025.10.mjs`, deleted after the run)
will apply these rules so the diff is mechanical and reviewable, rather than 62+
hand edits. It walks each `*.tokens.json`, converts by resolved `$type`
(respecting group-level `$type` inheritance), and preserves key order and all
`$description` text.

## 3. Files touched

**Token source (mechanical, via codemod):**
- `tokens/primitives/color.tokens.json` — color objects
- `tokens/semantic/color.tokens.json` — color objects (only literal hex; aliases untouched)
- `tokens/brands/*.tokens.json` — color objects (brand override hex)
- `tokens/components/*.tokens.json` — color objects where hex literals exist
- `tokens/primitives/spacing.tokens.json`, `radius.tokens.json`, `icon.tokens.json`, `letter-spacing.tokens.json`, `typography.tokens.json` — dimension objects
- `tokens/primitives/motion.tokens.json` — duration objects
- `tokens/primitives/shadow.tokens.json` — shadow sub-value objects

**Tooling:**
- `package.json` — `style-dictionary` `^4.0.0` → `^5`; reconcile any v4→v5
  config breakage (verify `transformGroup: 'css'`, `format: 'css/variables'`,
  `outputReferences`, `usesDtcg` against the v5 migration guide).
- `scripts/generate-docs.mjs` — **`generate-docs.mjs:68`** reads `token.$value`
  as a string for color swatches (`.toUpperCase()`, `.startsWith('#')`). Update
  to read `token.$value.hex` (with a components→hex fallback). Audit the rest of
  the file for any dimension/string `$value` assumptions.

**Unaffected (verified):**
- `scripts/validate.mjs` — reference resolver walks `Object.values`; object
  values pass through. Reference-integrity check still holds.
- `scripts/rules.mjs`, `scripts/drift-lint.mjs` — scan *consumer* CSS/TS, not
  token JSON. The new `hex` fallback lives only in source, never scanned.
- `scripts/build-design-system-json.mjs`, `schemas/meta.schema.json` — component
  metadata, not token format.
- `packages/tokens`, `portfolio-vercel` — consume built CSS; output is unchanged
  (see §4 gate).

## 4. Verification gate — build output parity

The migration is only accepted if the built CSS is byte-identical (modulo
incidental formatting):

1. Before any change: `npm run build:all` and snapshot `build/css/*.css`.
2. After SD v5 bump + token conversion: rebuild and `git diff` the snapshot.
3. **Expected diff: none.** Any real change to a `--token` value is a migration
   bug and blocks the phase.
4. Run `npm run validate` (references + meta) and `npm run docs`; confirm
   `docs/index.html` still renders swatches.
5. Confirm `npm run build:meta` artifact is unchanged (component metadata is
   untouched, so `design-system.json` should not move).

## 5. Commit sequence (on `claude/dtcg-token-standards-update-ge1ruy`)

1. `chore: snapshot build/css baseline for DTCG 2025.10 parity check` (temporary
   reference artifact, removed in the final commit)
2. `build: upgrade Style Dictionary to v5` — package bump + config reconcile +
   lockfile; prove base build still produces identical CSS.
3. `feat(tokens): migrate color values to DTCG 2025.10 object form`
4. `feat(tokens): migrate dimension/duration values to DTCG 2025.10 object form`
5. `feat(tokens): migrate shadow composite sub-values to object form`
6. `fix(docs): read object color $value in generate-docs`
7. `docs: record DTCG 2025.10 value-format adoption` — single dated entry in
   `docs/decisions.md` (what / why / alternative / status); note the new color
   format in `ai/DESIGN.md` (author-format note only — resolved tables stay).
   Remove the baseline snapshot.

Each commit rebuilds and keeps the §4 parity gate green.

## 6. Risks & rollback

- **SD v4→v5 config breakage** is the main unknown — it's a major version with
  its own breaking changes (Node/ESM/transform renames). Commit 2 isolates it so
  a problem there is contained and revertable on its own.
- **Float precision in `components`** could perturb CSS output if SD recomputes
  color from components instead of using `hex`. The parity gate (§4) catches
  this; if it triggers, pin SD to emit from the `hex` fallback.
- **Rollback:** each step is an independent commit; revert in reverse order. No
  consumer republish happens in Phase 1, so nothing downstream needs unwinding.

## 7. Explicitly deferred (Phase 2, not now)

- **Resolver module** — rewriting `build-brands.mjs` + brand files into resolver
  documents (sets/modifiers/contexts). Orthogonal to value encoding; enables
  light/dark modes later.
- **`$deprecated`** group/token property — aligns with the deprecated-token rule
  already in `scripts/rules.mjs`.
- **Wide-gamut color** (Display P3 / Oklch) — Phase 1 stays sRGB; revisit if a
  brand needs gamut beyond sRGB.
